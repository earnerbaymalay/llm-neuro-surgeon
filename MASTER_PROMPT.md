## §1 PRODUCT VISION (expanded, authoritative spec)

**Problem.** Power users of AI coding tools maintain the same knowledge many times: skills for Claude Code, GEMINI.md for Gemini CLI, AGENTS.md for Codex/OpenCode, .cursor/rules, .clinerules, CONVENTIONS.md for Aider, copilot-instructions.md, plus per-tool MCP server configs. Formats drift, skills go stale in some tools, and no single place shows what any given model "knows."

**Solution.** A local-first desktop app + CLI that performs "neurosurgery" on the user's machine: it scans for every AI tool's config, extracts skills/agents/rules/memory/prompts/MCP servers into ONE canonical, human-readable directory (**the Brain**), then projects that Brain back out to every tool via symlinks or generated files, kept in sync automatically. Result: every model is equally skilled; the user edits in one place, forever.

**Product pillars (all are in scope):**

1. **The Brain** — one central directory (default `~/AIBrain`, user-configurable), clearly laid out, git-backed.
2. **Universal Import** — adapters that scan and losslessly ingest configs from: Claude Code/Desktop, Gemini CLI, OpenAI Codex CLI, Cursor, Windsurf, Cline/Roo Code, Aider, Continue, GitHub Copilot instructions, Zed, OpenCode, VS Code MCP configs. (Verify each tool's *current* config paths/formats from official docs at build time — do not trust memorized paths.)
3. **Projection Engine** — per-tool output: symlink where the tool tolerates it; otherwise emit a generated file stamped `GENERATED BY <brand> — edit in the Brain`. Embrace the emerging `AGENTS.md` standard as a first-class projection target. Windows: directory junctions, hardlink or copy+watch fallback when symlink privileges are absent.
4. **Auto-Sync Daemon** — filesystem watcher + scheduled sweeps (launchd/systemd/Task Scheduler). Bidirectional: source-edited → re-import; Brain-edited → re-project. Both changed → three-way merge against last git snapshot; disjoint markdown merges auto-resolve, real conflicts enter a review queue in the GUI. Every sync = a git commit (the "Time Machine": browse, diff, roll back).
5. **Marketplace Import** — one-click import of skills/agents from: the official `anthropics/skills` GitHub repo, awesome-claude-code / awesome-ai-skills lists, Gemini CLI extensions & official docs examples, and any git URL or local folder. Show provenance, license, and a diff preview before install.
6. **MCP Hub** — browse/search plug-and-play MCP servers from the official MCP registry, Smithery, PulseMCP, mcp.so, and the Docker MCP catalog; install once, project the server config to every tool's MCP file; health-check (spawn + handshake) each server; secrets stored in the OS keychain and written to tool configs as env placeholders only.
7. **Vitals & Doctor** — dashboard showing the coverage matrix (tool × capability), drift/broken-link detection, and a "Doctor" diagnostic that explains and fixes issues ("Cursor lost 3 rules — reattach?"). This is the neurosurgeon persona: precise, calm, a little clinical.
8. **Safety by design** — dry-run is the default for the first merge; a full backup snapshot precedes any destructive op; import never follows symlinks outside scanned roots; imported skills are flagged as untrusted prompt/code (show source + checksum, require explicit enable); no telemetry; fully offline except explicit marketplace fetches.

**Non-goals (v1):** running models, editing chat histories, cloud sync, mobile.

**Success criteria:** fresh machine → onboarded Brain in <5 min; round-trip import/export is semantically lossless for all 12 adapters; a skill edited in the Brain appears in every linked tool within 5 s (watcher) or one schedule tick; app runs on macOS, Windows, Linux.

**Canonical Brain layout (build to this):**

```
AIBrain/
  skills/<slug>/        SKILL.md + skill.yaml (id, version, triggers, targets, source, sha256)
  agents/<slug>.md      canonical agent defs (frontmatter: tools, model hints, targets)
  rules/                global.md + scoped/<glob>.md   (the single source for all *.md rule files)
  memory/               MEMORY.md + topic files
  prompts/              reusable commands/templates
  mcp/servers/<id>.yaml (transport, command/url, env-placeholders, targets, health)
  .brain/               mappings.json (source↔canonical↔projection + hashes), state.json,
                        locks, logs/, backups/
  .git/                 full history
```

## §2 OPERATING SYSTEM — file-state, loop, and communication protocol

All memory lives in files, never in conversation. Create at start and keep current:

- `PLAN.md` — every task as `- [ ] T<phase>.<n> <task> :: verify: <command/check>`; gates tagged `[GATE]`.
- `PROGRESS.md` — append-only; 3 lines max per completed task (what, evidence, next).
- `DECISIONS.md` — every non-trivial technical decision, one line + rationale.
- `DESIGN_PACK.md` — created in Phase 1; binding for all UI work.
- `QUESTIONS.md` — deferred non-blocking questions, surfaced only at gates.

**RALPH LOOP (primary execution mode from Phase 2 onward).** Outer loop (human or script runs from terminal): `while PLAN.md has unchecked non-GATE tasks: claude -p "$(cat RALPH_PROMPT.md)"`. You will write `RALPH_PROMPT.md` in Phase 1 with exactly this inner algorithm:

1. Read `PLAN.md` + last 20 lines of `PROGRESS.md` (+ `DESIGN_PACK.md` only if the task is UI).
2. Select the SINGLE highest-priority unchecked, unblocked task.
3. Implement it completely.
4. Run its `verify:` check. Fix-and-rerun up to 3 attempts; still red → mark `[BLOCKED]` with a one-line note and stop touching it.
5. `git commit` (conventional message), tick the checkbox, append PROGRESS entry. STOP.

Each iteration starts with fresh context — that, not summarization, is the core token saver. Guard: if the last 3 PROGRESS entries reference the same task, mark it BLOCKED and escalate to `QUESTIONS.md`. The loop must halt at `[GATE]` tasks and request human review.

**Milestone updates.** At each gate, and only at gates, post:

```
## MILESTONE <n> — <phase>
Shipped: …            Evidence: <test counts, screenshot paths, demo cmd>
Risks: …              Decision needed: <one question + recommended option>
```

## §3 SWARM PROTOCOL — subagents, models, token economy

Use subagents aggressively but purposefully. Standing roster (create as project agents in `.claude/agents/`):

| Agent | Model | Job | Return contract |
|---|---|---|---|
| `recon` | Haiku | Verify current config paths/format of ONE tool from official docs; write `docs/research/<tool>.md` | ≤120-word brief |
| `adapter-smith` ×N | Sonnet | Build+test ONE adapter in an isolated git worktree; N run in parallel | ≤150 words + file list |
| `ui-builder` | Sonnet | Implement one screen strictly from `DESIGN_PACK.md` | ≤150 words + screenshot path |
| `test-writer` | Sonnet | Golden fixtures, round-trip property tests | ≤100 words |
| `red-team` | Opus | Attack phase-exit claims: symlink escapes, path traversal, destructive-op guards, junk configs | pass/fail + repro steps |
| `doc-writer` | Haiku | README, user guide, in-app copy | ≤80 words |

Orchestrator (you, Opus) only plans, integrates, reviews, and gates — never grinds boilerplate.

**Token economy — hard rules:**

1. File-state over chat-state; fresh ralph iterations instead of long sessions.
2. Grep first, then `Read` with offset/limit; never re-read unchanged files; never `cat` whole trees.
3. Batch all independent tool calls in one block; zero narration between calls.
4. Model tiering: Opus plans/reviews, Sonnet builds, Haiku researches/documents.
5. Subagent returns are capped (table above); details go to files, not the transcript.
6. Keep `RALPH_PROMPT.md` and `CLAUDE.md` byte-stable across iterations (prompt-cache friendly); `CLAUDE.md` ≤60 lines.
7. Reuse templates/skills; scaffold with generators, not token-generated boilerplate.
8. `/compact` (or equivalent) at every phase boundary; parallelize only independent tasks.
9. One adapter = one worktree = one subagent; merge via PR-style review by red-team, not re-reading diffs in main context.
10. Screenshots and test logs are referenced by path, never inlined.

## §A Target model & justification

**Target: Claude Opus 4.8 (`claude-opus-4-8`) running inside Claude Code**, with Sonnet 5 and Haiku 4.5 as subagent workers (the prompt's §3 table assigns them).

Why Opus 4.8 specifically: (1) this is long-horizon agentic engineering — hours of autonomous multi-file work where Opus-class models hold intent and drift least; (2) the plan depends on Claude Code-native machinery — subagents/swarm, skills, hooks, and headless `claude -p` for the ralph loop — and Opus 4.8 is the flagship tuned for exactly that harness; (3) the token strategy assumes an orchestrator/worker split: Opus only plans, reviews, and gates while cheaper models grind, which is both the fastest and the cheapest configuration. If your plan includes a Fable/Mythos-tier model, use it for Phase 0–1 (design judgment) and gate reviews, and keep Opus/Sonnet in the loop for build economics.
